<interfaces>
Target schema and API contract for this plan.

Required new tables and exact columns:

1. `lists`
- `id` INTEGER PK
- `professor_id` INTEGER NOT NULL FK `users.id`
- `title` VARCHAR(255) NOT NULL
- `deadline` TIMESTAMP NOT NULL
- `published` BOOLEAN NOT NULL DEFAULT FALSE
- `published_at` TIMESTAMP NULL
- `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP

2. `list_questions`
- `id` INTEGER PK
- `list_id` INTEGER NOT NULL FK `lists.id`
- `question_id` INTEGER NOT NULL FK `questions.id`
- `order_index` INTEGER NOT NULL
- unique(`list_id`, `question_id`)
- unique(`list_id`, `order_index`)

3. `list_submissions`
- `id` INTEGER PK
- `list_id` INTEGER NOT NULL FK `lists.id`
- `student_id` INTEGER NOT NULL FK `users.id`
- `started_at` TIMESTAMP NULL
- `submitted_at` TIMESTAMP NULL
- `correct_count` INTEGER NOT NULL DEFAULT 0
- `total_questions` INTEGER NOT NULL DEFAULT 0
- `score_pct` NUMERIC(5,2) NOT NULL DEFAULT 0
- `is_late` BOOLEAN NOT NULL DEFAULT FALSE
- unique(`list_id`, `student_id`)

4. `list_submission_answers`
- `id` INTEGER PK
- `submission_id` INTEGER NOT NULL FK `list_submissions.id`
- `question_id` INTEGER NOT NULL FK `questions.id`
- `selected_answer` VARCHAR(1) NULL
- `is_correct` BOOLEAN NOT NULL

5. `list_change_log`
- `id` INTEGER PK
- `list_id` INTEGER NOT NULL FK `lists.id`
- `professor_id` INTEGER NOT NULL FK `users.id`
- `action` VARCHAR(40) NOT NULL
- `summary` TEXT NOT NULL
- `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP

Status semantics that the service layer MUST enforce:
- `status`: `rascunho` when `published=false`; `publicada` when `published=true` and `deadline >= agora`; `encerrada` when `published=true` and `deadline < agora`
- `can_submit`: `true` for any published list, including `status='encerrada'`
- `student_status`: `nova`, `em_andamento`, `entregue`, `entregue_fora_do_prazo`

Required endpoints and payload shapes:

- `POST /lists`
  Request: `{ "title": string, "deadline": string, "question_ids": number[] }`
  Response 201: `{ "id": int, "status": "rascunho", "question_count": int }`

- `GET /lists`
  Professor-only list index with summary fields per list

- `GET /lists/<id>`
  Professor-only detail with metadata, ordered question IDs, and change log

- `PUT /lists/<id>`
  Professor-only update for title, deadline, and ordered question IDs

- `POST /lists/<id>/publish`
  Professor-only publish action; response includes computed status

- `GET /lists/assigned`
  Student feed with cards for home screen, including `status`, `student_status`, `deadline`, `submitted_at`, `score_pct`, `question_count`, `can_submit`

- `GET /lists/<id>/questions`
  Ordered quiz payload for a student; response must match `QuizInProgressScreen` question shape:
  `{ id, issue, alternatives, correct_answer, solution, image_q, image_s, difficulty, chapter_name, chapter_number, topic_name, source }`

- `POST /lists/<id>/start`
  Student-only idempotent start marker; creates or updates `started_at`

- `POST /lists/<id>/submit`
  Student-only submission endpoint. Request:
  `{ "responses": [{ "question_id": int, "selected_answer": string|null, "is_correct": boolean }], "correct_count": int, "total_questions": int }`

- `GET /lists/<id>/me`
  Student-only result summary for already submitted lists

- `GET /lists/<id>/results`
  Professor-only results payload:
  `{ "summary": {...}, "students": [...], "per_question": [...], "change_log": [...] }`
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| professor frontend -> `/lists` mutate routes | Professor-controlled payload can attempt to edit other professors' lists or inject invalid question IDs |
| student frontend -> `/lists/<id>/submit` | Student-controlled answers and question IDs can be tampered with before submission |
| backend -> analytics payloads | Results endpoint aggregates student data and must not leak lists to non-owning professors or students |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege | Professor list mutation | mitigate | Service layer verifies `professor_id` ownership on every read/write endpoint before executing repository mutations |
| T-04-02 | Tampering | Student submission payload | mitigate | `submit_list_service()` must reject any `question_id` not present in the list and recompute `correct_count` from validated answers only |
| T-04-03 | Information Disclosure | Results endpoint | mitigate | `GET /lists/<id>/results` is professor-only and must verify the requesting professor owns the list before returning student-level data |
| T-04-04 | Repudiation | Published-list edits | mitigate | Every publish/update after creation writes `list_change_log` rows with action, summary, professor_id, and timestamp |
</threat_model>
