<objective>
Build the professor management/results screen so published lists are reviewable through a summary-first experience with drill-down into students, per-question error rates, and edit history.

Purpose: after professors publish lists and students submit them, the professor needs a single place to see overall completion, identify who has or has not submitted, and spot which questions are generating the most errors.

Output: a real `ListManager` screen and a reusable `ListResultsPanel` component for summary metrics and drill-down data.
</objective>

  ```
      "assigned_students": number,
      "submitted_students": number,
      "average_score_pct": number,
      "highest_error_rate_pct": number

    "students": [
      {
        "student_id": number,
        "student_name": string,
        "student_status": "nova" | "em_andamento" | "entregue" | "entregue_fora_do_prazo",
        "submitted_at": string | null,
        "score_pct": number | null
      }
    ],
    "per_question": [
      {
        "question_id": number,
        "order_index": number,
        "error_rate_pct": number,
        "response_count": number
      }
    ],
    "change_log": [
      {
        "id": number,
        "action": string,
        "summary": string,
        "created_at": string
      }
    ]
  }
  ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| professor UI -> results payload | Detailed student data and change history must belong only to the selected professor-owned list |
| list index -> detail panel | Switching between lists must not leave stale results from the previously selected list on screen |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Information Disclosure | Results panel | mitigate | Results fetch is keyed by selected `listId`, and the backend already enforces professor ownership before returning student rows |
| T-04-12 | Tampering | Stale selected-list panel | mitigate | Reset loading/detail state whenever `listId` changes and refetch the full `/lists/<id>/results` payload |
| T-04-13 | Repudiation | Post-publication edits | mitigate | Always render the backend `change_log` block in the same results experience so later score shifts have visible provenance |
</threat_model>
