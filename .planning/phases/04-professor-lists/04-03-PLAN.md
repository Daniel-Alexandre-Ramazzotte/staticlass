<objective>
Build the student list workflow on top of the existing home, quiz, and result screens: assigned lists on the home tab, list-aware quiz playback, submission recording, and late-state result viewing.

Purpose: students should discover lists from the home screen, answer them through the same quiz flow they already know, and see whether they submitted on time or late without the app splitting into a separate list-only UX stack.

Output: `Minhas Listas` on home, list-mode branches in `QuizInProgressScreen` and `ResultScreen`, and a read-only result path for already submitted lists.
</objective>

<interfaces>
Required backend payloads from Plan 04-01:

- `GET /lists/assigned`
  Response item contract:
  `{ id, title, deadline, status, student_status, question_count, submitted_at, score_pct, can_submit }`

- `GET /lists/<id>/questions`
  Quiz payload shape must match the existing `QuizQuestion` interface in `QuizInProgressScreen.tsx`

- `POST /lists/<id>/start`
  Response 200: `{ "message": "lista iniciada" }`

- `POST /lists/<id>/submit`
  Request:
  `{ "responses": [{ "question_id": number, "selected_answer": string|null, "is_correct": boolean }], "correct_count": number, "total_questions": number }`
  Response:
  `{ "student_status": "entregue" | "entregue_fora_do_prazo", "submitted_at": string, "score_pct": number, "is_late": boolean }`

- `GET /lists/<id>/me`
  Response:
  `{ id, title, deadline, submitted_at, score_pct, student_status, is_late, correct_count, total_questions }`

Route-param contract to implement:
- `home.tsx` routes to `QuizInProgressScreen` with `{ list_id, list_title, list_mode: "1" }` for not-started/in-progress lists
- `home.tsx` routes to `ResultScreen` with `{ list_id, list_title, submission_view: "1" }` for already submitted lists
- `QuizInProgressScreen` passes `list_id`, `list_title`, and `list_mode` onward to `ResultScreen`

Status-to-CTA mapping:
- `nova` -> `Começar`
- `em_andamento` -> `Continuar`
- `entregue` -> `Ver resultado`
- `entregue_fora_do_prazo` -> `Ver resultado`
- `status="encerrada"` and not submitted -> `Entregar atrasada`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| aluno home -> assigned-list feed | Student-facing home cards expose deadline and score metadata that must come only from authenticated backend responses |
| quiz -> submit result | Serialized `resultado` can be tampered with on the client before submission |
| result view -> read-only fetch | Existing result screen must not re-submit an already completed list when opened in view mode |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Tampering | Client-built `responses` payload | mitigate | Backend submit endpoint validates that question IDs belong to the list and recomputes score from per-answer correctness |
| T-04-09 | Repudiation | Re-opening a result screen | mitigate | `submission_view === "1"` branch performs a read-only fetch and never replays a submit call |
| T-04-10 | Information Disclosure | Aluno home cards | accept | feed comes from `/lists/assigned` under JWT auth and only returns lists assigned to the current aluno |
</threat_model>
