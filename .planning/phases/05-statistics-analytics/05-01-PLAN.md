<objective>
Create the canonical answer-history foundation for Phase 5 so every new free-practice session and every list submission produces durable per-answer analytics rows.

Purpose: the student dashboard, professor risk views, and admin KPI screens in this phase must stop depending on aggregate-only quiz history. This plan establishes `answer_history`, backfills only exact historical list answers, and wires both quiz-completion flows to write answer-level data at the moment the user finishes a session.

Output: one new answer-history model/repository pair, incremental schema bootstrap for the new table/indexes, updated free-practice and list-submission write paths, and backend tests proving the canonical rows are created.
</objective>

<interfaces>
Canonical persistence and request contracts for this plan.

Required new table:

1. `answer_history`
- `id` INTEGER PK
- `student_id` INTEGER NOT NULL FK `users.id`
- `question_id` INTEGER NOT NULL FK `questions.id`
- `answered_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
- `is_correct` BOOLEAN NOT NULL
- `selected_answer` VARCHAR(1) NULL
- `source` VARCHAR(20) NOT NULL with only `free_practice` or `list`
- `source_id` INTEGER NOT NULL
- `list_id` INTEGER NULL FK `lists.id`
- unique(`source`, `source_id`, `question_id`)

Required indexes:
- `(student_id, answered_at DESC)`
- `(question_id, answered_at DESC)`
- `(list_id, answered_at DESC)`
- `(source, source_id)`

Free-practice session request update:

- `POST /gamification/record-session`
  Existing required fields remain:
  `{ "acertos": number, "total": number, "capitulo_id"?: number, "dificuldade"?: number }`
  Add required answer payload when a free-practice quiz is completed:
  ```
  {
    "acertos": number,
    "total": number,
    "capitulo_id"?: number,
    "dificuldade"?: number,
    "answers": [
      {
        "question_id": number,
        "selected_answer": string | null,
        "is_correct": boolean
      }
    ]
  }
  ```

List-submission analytics mirroring:

- `POST /lists/<id>/submit`
  Request shape stays the same for the client.
  Backend side effect is mandatory:
  validated `responses` must be mirrored into `answer_history` using:
  - `source = 'list'`
  - `source_id = list_submissions.id`
  - `list_id = lists.id`
  - `answered_at = list_submissions.submitted_at`

Historical backfill rules:
- Backfill only from `list_submissions` + `list_submission_answers`
- Do NOT synthesize answer rows from `quiz_resultados`
- Backfill must be idempotent across repeated app boots or retries

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| student client -> answer-history persistence | Client payload includes correctness and selected answers, but student identity and list identity must remain server-derived |
| list resubmission -> canonical analytics rows | Re-submitting the same list must replace the canonical rows for that submission instead of duplicating or orphaning old answers |
| historical bootstrap -> current runtime writes | One-time backfill must not overwrite newer rows or fabricate answer-level history that never existed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | Free-practice answer payload | mitigate | Validate answer count, question ids, and boolean correctness in `record_session_service()` and derive `student_id` from JWT instead of trusting request body identity fields |
| T-05-02 | Tampering | List answer-history sync | mitigate | Use route `list_id`, repository `submission_id`, and authenticated `student_id`; do not accept these values from the client payload |
| T-05-03 | Repudiation | List resubmission analytics | mitigate | Replace canonical rows inside the same transaction keyed by `(source, source_id, question_id)` so the latest submission state is auditable and deterministic |
| T-05-04 | Information Integrity | Historical backfill | mitigate | Backfill only exact list-answer rows from `list_submission_answers`; never infer free-practice answers from aggregate `quiz_resultados` totals |
</threat_model>
